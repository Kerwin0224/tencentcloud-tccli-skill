# Agent 优化模式深度分析

本文档是 `SKILL.md`「Agent 优化模式」节的补充——当 agent 需要理解**为什么**某个模式有效、或者在多个模式间做权衡时加载。日常调用只需看 SKILL.md 的速查表。

---

## agent 的消费约束

要判断 tccli 的哪些特性对 agent 友好，先要理解 agent 作为 API 消费者的独特约束：

1. **Token budget 稀缺**：上下文窗口有限，每个 token 都有成本和注意力负担
2. **推理强但怕噪声**：返回中大量无关字段（`RequestId`、嵌套元数据）稀释关键信息、增加幻觉风险
3. **多步编排**：上一步的输出喂给下一步，结果必须可链式流转
4. **长任务**：云操作大量异步（开机、快照、扩容），需要"等待完成"原语

基于这四条，agent 友好的特性 = 提高**信息密度** + 提升**确定性** + 保持**可组合**。

## 结构性缺陷

这些是 agent 最难以在外部自行弥补的能力缺口：

| 缺陷 | 影响 | 缓解方式 |
|------|------|---------|
| 退出码粗糙 (0/252/255) | agent 无法做失败分类决策 | **必须**解析 `Response.Error.Code`，退出码仅作辅助 |
| 无 `--dry-run` | 破坏性操作无预检机制 | 对写操作做 `--generate-cli-skeleton` + 人工确认；或先 `Describe` 确认目标 |
| `--waiter` 用 JSON 而非 Python 字面量 | 格式错误导致解析失败 | 严格用双引号 JSON 格式 |
| 无输出骨架 | 模板化只覆盖入参，输出结构靠猜 | 先用 `Limit=1` 最小查询学习输出结构 |
| 无请求级重试/退避 | 网络抖动一次就失败 | agent 在 CLI 外层实现退避重试（仅幂等操作） |
| 无结构化错误输出 | 错误文本随 `--language` 变化 | 用 `--language en-US` 锁定英文 |

## 组合模式

### 压缩管道（最常用）

```
--filter (JMESPath 裁剪) + --output text (去 JSON 结构)
```

本质：**用 CLI 的本地算力换 agent 的 token 预算**。数据变换从 agent 的推理层下沉到 CLI 的执行层。

### 模板驱动（复杂入参）

```
--generate-cli-skeleton → 编辑 JSON → --cli-input-json file:// → --waiter → --filter + output=text
```

四段式：模板获取 → 参数填充 → 异步调用 → 结果裁剪。每段独立，可部分复用。

### 审计链（多 agent 协作）

```
--profile prod --request-client "agent-name/v1.0"
```

CloudAudit 中可见完整链路：谁（request-client）以什么身份（profile）做了什么操作。`(request-client, profile, RequestId)` 三元组唯一定位一次调用。

### 自举学习（API 探索）

```
1. --generate-cli-skeleton 拿入参骨架
2. 最小查询（Limit=1）拿输出结构（替代缺失的输出骨架）
3. 基于输出结构构造 --filter 表达式
4. 后续调用走压缩管道
```

## agent 友好的 CLI 设计哲学

从 tccli 的分析中抽象出六条原则，可用于评估任何 CLI 对 agent 的友好程度：

1. **确定性优先**：同样输入 → 同样输出。默认 JSON，`--language` 锁定环境
2. **可组合性**：正交 flags 自由叠加，无需外部胶水代码
3. **信息密度可控**：agent 能按需取用，而非全有或全无
4. **幂等与可重试内建**：长任务原语（`--waiter`），理想状态还应有请求级重试
5. **模板化**：skeleton + input-json 让参数可复用、可验证
6. **可审计性**：`--request-client` 提供行为维度的责任归属

tccli 在 ①②③⑤⑥ 上表现优秀，在 ④ 上（重试/退避/幂等 token 自动注入）还有提升空间。
